Chapitre 01

Vue d'ensemble & écrans

Une application de gestion pour agences immobilières, pensée comme une démonstration commerciale : chaque écran met en scène une ou plusieurs automatisations.

ImmoMail Studio réunit dans une seule interface le quotidien d'une agence — leads, visites, mandats, baux, conformité, marketing — et montre, écran par écran, ce que les automatisations font à la place de l'humain.

L'interface

Le cadre commun à tous les écrans

  • Barre latérale (desktop) / menu tiroir (mobile) — navigation entre les écrans.
  • Sélecteur d'agence — bascule entre les agences ; toutes les données sont cloisonnées par agence.
  • Barre d'horloge de démo — affiche et fait avancer la date simulée pour déclencher en direct les automatisations différées (rappels J-1, avis J+2, parrainage J+30…).

Le modèle de rendu

Server Components + Server Actions

  • Lecture. Les pages sont des composants serveur : elles interrogent la base via Drizzle (lib/queries.ts) et rendent le HTML côté serveur — pas d'appel réseau client.
  • Écriture. Les interactions (réserver, importer, envoyer une newsletter) passent par des Server Actions (app/actions.ts).
  • Couche de services mockée. lib/services/* simule email, SMS, PDF, calendrier — branchable sur de vrais fournisseurs (ce que fait la couche n8n).
  • Contexte de page. lib/page-context.ts et lib/agency.ts résolvent l'agence sélectionnée pour cloisonner les données.

Chapitre 04

Automatisations & moteur

Onze automatisations pilotées par un moteur idempotent, déclenché par l'horloge de démo dans l'app — ou par n8n en production.

Le catalogue

4 familles, 11 automatisations

FamilleAutomatisations
LeadsA9 tri & qualification · A10 réponse instantanée · A11 fiche CRM
VisitesA1 prise de RDV · A2 confirmation + rappels
GestionA3 expiration de mandat · A4 quittances · A5 conformité
MarketingA6 newsletter · A7 avis Google · A8 parrainage

Le moteur

lib/automation-engine.ts

  • Un seul point d'entrée. runEngine(upto, agencyId?) passe en revue chaque agence et traite tout ce qui est « dû » jusqu'à la date upto.
  • Idempotence. Chaque action réserve une clé unique dans automation_runs.run_key — jamais deux fois le même envoi.
  • Traçabilité. Tout envoi est écrit dans messages + activity_log.

L'horloge de démo

lib/demo-clock.ts

La table demo_clock porte une date simulée. L'avancer (depuis la barre du haut) rejoue le temps et déclenche les automatisations différées — J-1 avant une visite, J+2 après une signature, J+30 pour le parrainage — en direct devant le prospect. En production, cette horloge est remplacée par le temps réel piloté par n8n.

Chapitre 05

Import / export & API

L'agence charge ses propres données via Excel, et la démo s'adapte immédiatement. Trois routes API portent le PDF, l'export et l'import.

Le round-trip Excel

lib/excel-io.ts (exceljs)

L'écran /import propose d'exporter un classeur modèle, de le remplir avec ses propres biens et contacts, puis de le réimporter : l'affichage reflète alors fidèlement le fichier. Le classeur couvre agences, biens, acheteurs, boîte de réception, leads, rendez-vous, mandats, baux, conformité, transactions et segments.

Les 3 routes API

RouteMéthodeRôle
/api/templateGETExporte l'agence sélectionnée en .xlsx (sert de modèle).
/api/importPOSTImporte un .xlsx : remet à zéro puis recharge les données.
/api/receipt/:idGETGénère la quittance de loyer en vrai PDF (pdf-lib), avec ?period=YYYY-MM.

La route quittance est aussi celle qu'appelle le workflow n8n A4 pour joindre le PDF à l'email — l'app et l'automatisation partagent le même générateur.

Chapitre 06

Installation & déploiement

De git clone à la production sur Vercel — avec le piège Supabase à ne pas rater.

Lancer en local

  1. Renseigner DATABASE_URL dans .env.local (chaîne Supabase, pooler Session, port 5432).
  2. Installer : npm install.
  3. Peupler les données de démo : npm run seed.
  4. Démarrer : npm run dev → l'app tourne, date de démo initiale au 23 juin 2026.

Les scripts

ScriptRôle
npm run devServeur de développement.
npm run build / startBuild de production / serveur.
npm run seedCharge les 3 agences et données de test (scripts/seed.ts).
npm run db:pushApplique le schéma Drizzle (drizzle-kit push).

Le piège Supabase + postgres-js

À retenir absolument

Utiliser le pooler Supabase en mode Session (port 5432), pas le mode transaction (6543) : avec postgres-js, le mode transaction bloque sur la première requête (récupération des types) → timeouts 504. Côté client : prepare:false, fetch_types:false, connexion mise en cache sur globalThis (réutilisation serverless). Mot de passe à caractères spéciaux → encoder en %xx dans l'URL.

Production

Vercel + Supabase EU

  • Hébergement : Vercel (projet immomail-studio), déploiement auto.
  • Base : Supabase région EU (RGPD) — à choisir à la création du projet, non modifiable ensuite.
  • Vigilance offre gratuite : un projet Supabase gratuit se met en pause après ~7 jours d'inactivité ; le site renvoie alors des erreurs 500. Le restaurer (bouton Restore) suffit à tout remettre en ligne.
